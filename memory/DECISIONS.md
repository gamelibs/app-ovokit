## 决策 001：5 种核心玩法原型作为第二层分类，与 12 母型并存

- **日期**：2026-06-14
- **问题**：如何从编辑器架构角度扩展站点的分类体系？
- **选项**：
  1. 用 5 种核心原型替代 12 母型
  2. 把 5 种核心原型作为独立维度与 12 母型并存
  3. 把 5 种核心原型仅用于 Demo 模板，不在站点导航中展示
- **决定**：选项 2 —— 12 母型（玩家感受） + 5 核心原型（编辑器架构）双层分类并存。
- **原因**：
  - 12 母型面向学习者，提供"这是什么玩法"的锚点
  - 5 核心原型面向实现者，提供"底层循环和数据结构是什么"的锚点
  - 两者互补，不互相替代
- **影响**：
  - `PlayMeta` 新增可选字段 `pattern`
  - 首页筛选新增"核心原型"分组
  - 母型详情页需要显示所属核心原型
  - 版主发帖表单增加核心原型选择

## 决策 002：5 种核心原型的具体定义

- **日期**：2026-06-14
- **问题**：5 种核心原型的边界如何划分？
- **决定**：
  1. **action** — 动作敏捷：Input → Avatar → Physics → Score
  2. **spatial** — 空间布局：Board → Cell → Rule → State Change
  3. **merge** — 合成成长：Resource → Merge → Level Up → Production
  4. **management** — 经营模拟：Building → Production → Economy → Growth
  5. **strategy** — 数值策略：Unit → Stats → Combat → Reward
- **原因**：这 5 种循环覆盖了绝大部分小游戏的编辑器数据结构；每个原型的抽象节点都对应可实现的 Demo 模板。
- **影响**：
  - Demo 模板按这 5 种循环设计
  - 版主选择原型后，自动生成对应循环的 breakdown 骨架

## 决策 003：先实现数据模型与筛选，再补 Demo

- **日期**：2026-06-14
- **问题**：实施 5 种核心原型的优先级如何安排？
- **决定**：分 3 个迭代：
  1. 迭代 1：数据模型 + 首页筛选 + 版主表单
  2. 迭代 2：母型映射 + 页面展示
  3. 迭代 3：5 个核心原型最小 Demo
- **原因**：让分类体系先 selectable/filterable，版主能先用起来；Demo 可以后续逐步补齐，不影响内容生产流程。
- **影响**：迭代 1 完成后，内部即可开始按核心原型发帖和筛选。

## 决策 004：核心原型元数据以代码常量为主，文案可后续迁移到 content/

- **日期**：2026-06-14
- **问题**：核心原型元数据放在代码里还是 content/ 目录？
- **选项**：
  1. 像母型一样放在 `content/patterns/<key>/meta.json`
  2. 放在 `src/lib/patterns/patterns.ts` 作为稳定常量
- **决定**：选项 2 作为第一阶段实现；后续如果版主需要在线编辑原型文案，再迁到 `content/patterns/`。
- **原因**：5 种核心原型是架构层的稳定分类，不频繁变动；代码常量实现更快，类型更安全。
- **影响**：当前无需新增 content 读取逻辑；后续迁移成本低。
