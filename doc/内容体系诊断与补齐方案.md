## 2. 发布工作流现状

### 2.1 当前路径（版主手动操作）

```
版主点击 Logo 8 次 → /mod/new
  ├── 填写基础信息（标题/副标题/slug/难度）
  ├── 选择标签（TagInput，27 个预定义）
  ├── 填写技术栈 / 核心点（TagInput）
  ├── 编辑玩法拆解（BreakdownEditor：章节+要点）
  ├── 编辑代码片段（CodeSnippetEditor：标题+语言+代码）
  ├── 上传封面 或 一键生成 SVG 封面
  ├── 填写 Demo 信息（iframeSrc / 视频上传 / 说明）
  ├── 编写文章 或 AI 生成文章（/api/generate-article）
  └── 提交 → 写入 content/plays/<slug>/meta.json + article.mdx
```

| 能力 | 状态 | 瓶颈 |

## 3. 用户愿景 vs 当前现实

### 3.1 愿景工作流

```
输入：10 / 100 / 1000 个游戏原始素材
  ↓
[AI 分析] → 玩法核心规则 / 状态机 / 数值公式

[AI 生成] → 流程图 SVG + 手绘封面 SVG + 代码片段 + 文章

[中间工具] → 参数化 Demo 引擎 → 可试玩 Demo
  ↓

  ↓
```

### 3.2 逐项对比

| 愿景环节 | 当前状态 | 差距 | 阻塞度 |
|----------|----------|------|--------|
| **AI 分析游戏素材** | 无 LLM 接入 | 需接入 OpenAI/Claude API + 设计 prompt 链 | 🔴 高 |

| **生成代码片段** | `/api/generate-article` 只拼接已有代码 | 需 AI 根据规则生成真实可运行代码 | 🔴 高 |
| **中间工具制作 Demo** | 每个 Demo 需手写 TS 组件或状态机 | Demo 系统架构不支撑参数化配置 | 🔴 **最高** |
| **批量发布** | 无批量接口 | 需脚本化/API 化创建 | 🟡 中 |
| **用户组合玩法** | 无组合机制 | 需新增"混合/变体"概念和数据模型 | 🟡 中 |

## 4. 核心缺失诊断（按优先级排序）

### 🔴 P0 — 架构级缺失（阻碍规模化）

#### 缺失 1：Demo 系统不可扩展
**现状**：
- `PlayMiniDemo.tsx` 是**硬编码 slug → React 组件**的映射表（560+ 行）
- `ServerDemoPlayer` + Fastify 状态机需要为每个 Demo 手写 `init()` + `step()` + Zod schema
- 新增 1 个 Demo 需要写 200-500 行 TypeScript

**后果**：
- 30 个内容已接近手工维护上限
- 100/1000 个素材完全不可想象
- AI 无法自动产出 Demo（因为需要手写代码）

**根因**：
当前 Demo 架构是**"为每个玩法定制开发"**，而非**"参数驱动通用引擎"**。

#### 缺失 2：无 AI 基础设施
**现状**：
- 没有接入任何 LLM API
- `/api/generate-article` 只是模板拼接，不是真正的 AI 生成
- 无 prompt 管理、无链式调用、无结果缓存

**后果**：
- 无法自动分析游戏素材
- 无法自动生成代码
- 无法自动生成流程图 DSL

#### 缺失 3：无流程图系统
**现状**：
- 站点没有任何流程图渲染能力
- `sketch-svg/generator.ts` 有 4 个流程图形状（flow-start/process/decision/end），但只是孤立形状，无连接/布局能力
- 文章 MDX 不支持 Mermaid

**后果**：
- 玩法核心循环无法可视化呈现
- AI 即使分析了流程，也无地方展示

### 🟡 P1 — 体验级缺失（影响效率）

#### 缺失 4：封面 100% 占位
- 全部 30 个 Play 使用同一张 SVG 占位图
- 站点视觉上缺乏辨识度

#### 缺失 5：无批量发布工具
- 只能单条手动创建
- 无 CLI 脚本、无 CSV/JSON 批量导入

#### 缺失 6：无玩法组合/变体机制
- Play 之间完全独立
- 用户无法通过参数混合两个玩法生成新 Demo

## 5. 补齐路线图

### Phase 1：AI 基础设施 + 批量发布（1-2 周）

**目标**：让"素材 → 结构化内容"这条路跑通

| 任务 | 具体工作 |
|------|----------|
| 接入 LLM | 在 `.env.local` 配置 `OPENAI_API_KEY` 或 `ANTHROPIC_API_KEY`；创建 `src/lib/ai/` 封装流式调用 |
| 创建 `/api/ai/analyze` | 接收：游戏描述/截图 URL；返回：`{ title, subtitle, tags, breakdown, corePoints, flowchartDsl }` |
| 流程图渲染引擎 | 基于 `sketch-svg` 扩展：读取流程图 DSL（文本格式）→ 自动布局 → 生成手绘 SVG |
| 封面自动匹配 | 根据标签自动选择模板类型（"消除"→ blocks，"塔防"→ tower，"合成"→ gem...） |
| 批量创建 API | `POST /api/mod/plays/batch` 接收 JSON 数组，循环写入 |
| 批量 CLI 脚本 | `scripts/bulk-import-plays.ts`：读取 `imports/` 目录下的 JSON，调用本地 API 创建 |

### Phase 2：参数化 Demo 引擎（2-3 周）

**目标**：把 Demo 从"手写代码"变成"配置驱动"

**核心设计**：把玩法抽象为有限的"原子机制"

```typescript
// 玩法配置示例（AI 可生成）
const demoConfig = {
  type: "grid",           // grid | physics | merge | wave | choice | dodge
  params: {
    width: 9,
    height: 9,
    obstacles: [[3,3], [4,3], [5,3]],
    playerStart: [1,1],
    winCondition: "reach",
    target: [7,7],
  },
  controls: ["arrowKeys", "buttons"],
};
```

**引擎架构**：
```
DemoEngine（通用 Canvas 渲染器）
  ├── GridWorldRenderer    // 网格类：移动/碰撞/消除/合成
  ├── PhysicsWorldRenderer // 物理类：切割/弹射/堆叠
  ├── WaveSpawnerRenderer  // 波次类：塔防/生存
  ├── ChoiceTreeRenderer   // 选择类：策略/Roguelike
  └── StateMachineRenderer // 状态类：战斗/FSM
```

每个 Renderer 接收配置对象，不需要手写代码。

**实现步骤**：
1. 把 `PlayMiniDemo` 中的 4 个硬编码 Demo 抽象为 `GridWorldRenderer` 和 `StateMachineRenderer`
2. 新增配置解析层：`demoConfig → renderer + params`
3. 在 `meta.json` 中新增 `demoConfig` 字段
4. `embed/plays/[slug]` 改为读取 `demoConfig` 并交给对应 Renderer

### Phase 3：玩法组合系统（2-3 周）

**目标**：让用户可以混合两个玩法生成变体

**核心设计**：
```typescript
// 组合配置
const hybrid = {
  base: "td-waves-and-ai-scaling",
  mixin: "roguelike-horde-density-and-readability",
  overrides: {
    "wave.budgetMultiplier": 1.5,
    "enemy.pool": ["minion", "elite", "boss"],
  }
};
```

**前端交互**：
- 在详情页底部新增"混合此玩法"按钮
- 选择第二个玩法 → 显示可混合参数 → 生成预览 → 保存为新 Play

## 6. 即时可执行的小任务（本周可做）

以下任务不需要大架构改动，可以快速提升内容质量：

### 任务 A：封面去占位（1-2 天）
- 用 `scripts/generate-svg-assets.ts` 的思路，为每个 Play 根据标签生成专属封面
- 例如："消除"→ blocks 模板 + 标题文字；"塔防"→ tower 模板 + 标题文字
- 批量写入 `public/plays/<slug>/cover.svg`

### 任务 B：Demo 回退优化（半天）
- 当前无 iframeSrc 的 24 个 Play 回退到母型 Demo
- 但母型 Demo 的 `demoId` 映射是写死在 `embed/demos/archetype/[key]/page.tsx` 中的
- 改为从 `archetypes.ts` 动态读取，减少硬编码

### 任务 C：AI 文章生成增强（半天）
- 当前 `/api/generate-article` 只是拼接
- 可接入 LLM，在拼接基础上增加"玩法设计洞察"段落（让文章更有深度）

## 7. 结论

> **当前站点的内容体系对于 30 条内容是可以运转的，但对于用户设想的"10/100/1000 条素材 + AI 全自动工作流"，存在 3 个架构级缺失：**
>
> 1. **Demo 系统不可扩展**（最大瓶颈）—— 当前是手写代码模式，必须重构为参数化引擎
> 2. **无 AI 基础设施** —— 需接入 LLM API 并设计 prompt 链
> 3. **无流程图系统** —— 需新增手绘风格流程图 DSL + 渲染引擎
>
> **建议优先级**：
> - 如果目标是**先让现有 30 条内容达到"可对外展示"的质量** → 做任务 A（封面去占位）+ 接入 LLM 做文章增强
> - 如果目标是**支撑 100+ 条素材的规模化工作流** → 必须先做 Phase 2（参数化 Demo 引擎），这是地基
