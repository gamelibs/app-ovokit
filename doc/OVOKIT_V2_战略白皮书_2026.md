## 一、执行摘要

OVOKIT 已从"手绘风格小游戏展示站"升级为**游戏玩法知识库平台（Game Design Knowledge Base）**。在 2026 年 AI 生成内容（AIGC）全面渗透创作领域的背景下，OVOKIT 的核心竞争力不再是 UI 美观度，而是：**能否建立一套可扩展、可自动化、可组合的玩法知识体系**。

**本白皮书定义 OVOKIT V2 的五大战略支柱：**

1. **知识体系化** — Archetype → Play → Demo → Algorithm → Source 的五层内容模型
2. **内容工业化** — AI Agent 流水线 + 结构化 Schema，实现玩法内容的规模化生产
3. **体验闭环化** — 30秒认知 → 3分钟理解 → 10分钟实现 → 试玩验证 → 收藏复用
4. **技术解耦化** — 主站（Next.js）与 Demo 实验室（独立部署）分离，各尽所长
5. **增长有机化** — SEO 原生优势 + 知识型内容壁垒，自然流量驱动长期增长

## 二、时代背景：2026 年的 AI 与游戏开发

### 2.1 AIGC 已重塑内容生产

2026 年，AI 生成代码、AI 生成游戏原型、AI 生成设计文档已成为独立开发者的日常工具。但一个核心矛盾日益尖锐：

> **AI 能生成代码，但 AI 不懂"为什么这样设计"。**

ChatGPT/Claude 可以写出 functioning 的 Match-3 逻辑，但无法解释：
- 为什么 Candy Crush 的连锁反馈要控制在 3 秒内？
- 为什么 2048 的随机生成需要"软控难度"而非真随机？
- 为什么 Roguelike 的 Build 多样性比单局平衡更重要？

**这就是 OVOKIT 的价值锚点：沉淀"设计思想"，而非仅仅是"实现代码"。**

### 2.2 独立开发者成为核心用户群

Steam 每年新增 1.5 万款游戏（2025 年数据），其中 70% 来自独立开发者。他们的核心痛点：

| 痛点 | 现有解决方案的缺陷 |
|------|------------------|
| 想做一个 Tower Defense，不知道核心循环怎么设计 | 教程分散在 YouTube/博客，搜索成本高 |
| 看到一个好玩法，想快速验证原型 | 需要自己从零写 Demo，周期长 |
| 做完一个玩法，不知道还能组合出什么 | 缺乏"玩法组合引擎" |
| 代码能跑，但体验不对 | 缺乏"设计陷阱"的系统性总结 |

OVOKIT V2 的定位正是解决这些痛点。

### 2.3 搜索引擎偏爱知识型内容

Google 2024-2025 年的算法更新（Helpful Content Updates + AI Overviews）明确传递信号：

> **深度、结构化、可验证的知识内容获得更高权重。**

纯游戏导航站（只列链接+截图）的 SEO 价值正在衰减。而 OVOKIT 的"玩法拆解 + 代码 + Demo"三位一体内容，天然符合 Google 对 E-E-A-T（经验、专业、权威、可信）的要求。

**预测性判断：到 2027 年，"match 3 mechanics"、"tower defense design patterns" 等关键词的搜索量将增长 200%+，而 OVOKIT 有能力占据这些长尾词的首页位置。**

## 三、产品定位：不是导航站，是知识库

### 3.1 定位声明

```
OVOKIT = Game Design Knowledge Base

为独立游戏开发者和学习者提供：
├─ 玩法发现（Discovery）
├─ 玩法理解（Comprehension）
```

### 3.2 与竞品的差异化矩阵

| 维度 | 小游戏导航站 | 游戏资讯站 | 教程博客 | **OVOKIT V2** |
|------|------------|----------|---------|--------------|
| 内容深度 | 浅（只列游戏） | 浅（新闻报道） | 中（单篇教程） | **深（系统拆解）** |
| 结构化程度 | 无 | 无 | 低 | **高（Schema 驱动）** |
| 可试玩性 | 有（外链） | 无 | 无 | **有（嵌入式 Demo）** |
| 代码参考 | 无 | 无 | 部分有 | **有（算法 + 源码）** |
| 组合探索 | 无 | 无 | 无 | **有（玩法公式库）** |
| AI 扩展性 | 无 | 无 | 低 | **高（Agent 流水线）** |

### 3.3 用户模型与场景

#### 用户 A：游戏玩家（占比 ~40%）
- **目标**：了解某类玩法，寻找新游戏灵感
- **行为路径**：搜索 → 浏览母型 → 试玩 Demo → 离开
- **停留时间**：30 秒 ~ 3 分钟
- **价值**：流量基础，SEO 验证

#### 用户 B：独立开发者（占比 ~45%）
- **目标**：学习玩法设计，验证原型思路
- **行为路径**：搜索 → 阅读拆解 → 查看算法 → 试玩 Demo → 收藏
- **停留时间**：5 ~ 20 分钟
- **价值**：核心用户，内容消费主力

#### 用户 C：程序员（占比 ~15%）
- **目标**：直接实现某玩法机制
- **行为路径**：搜索算法 → 看代码片段 → 复制实现 → 调试
- **停留时间**：10 ~ 60 分钟
- **价值**：技术背书，口碑传播者

## 五、内容模型：五层知识体系

### 5.1 核心架构

```
Archetype（母型玩法）
    │
    ├─ Play（具体玩法）
    │       ├─ Demo（最小可试玩实现）
    │       ├─ Article（设计文章）
    │       └─ Source（代码实现）
    │
    └─ Algorithm（通用算法库）
            ├─ 匹配检测
            ├─ 路径规划
            └─ 状态机
```

**设计原则**：每一层都有明确的职责边界，层与层之间通过 Schema 关联。

### 5.2 Archetype（母型）— 认知锚点

母型不是分类标签，而是**"玩法基因"**。

当前 12 个母型：
- Match-Clear（消除）
- Dodge-Avoid（躲避）
- Runner（跑酷）
- Shoot-Aim（射击）
- Combat（战斗）
- Placement（放置/建造）
- Choice-Strategy（策略）
- Physics（物理）
- Puzzle（解谜）
- Progression（成长/数值）
- Simulation（模拟）
- Timing（时机/反应）

**V2 扩展方向**：
- 每个母型定义"核心机制 DNA"（输入 → 判定 → 反馈 → 状态更新）
- 母型之间定义"组合规则"（消除 + 战斗 = Puzzle RPG）

### 5.3 Play（具体玩法）— 知识单元

Play 是 OVOKIT 的核心内容单元。V2 要求每个 Play 必须包含以下字段：

```typescript
interface PlaySchema {
  slug: string;           // URL 标识
  title: string;          // 玩法名称
  archetype: string;      // 所属母型
  difficulty: "入门" | "进阶" | "硬核";
  
  // Section 1: 认知锚点
  summary: string;        // 30秒简介
  coreLoop: string;       // 核心循环一句话
  
  // Section 2: 系统拆解
  breakdown: {
    playerGoal: string;
    decision: string;
    feedback: string;
    loop: string;
  }[];
  
  // Section 3: 玩法公式
  formula: string;        // 例如："消除 = 匹配 + 反馈 + 状态更新"
  combos: string[];       // 可组合的方向
  
  // Section 4: 设计陷阱
  pitfalls: {
    title: string;
    description: string;
    solution: string;
  }[];
  
  // Section 5: 实现参考
  algorithms: {
    title: string;
    approach: "DFS" | "BFS" | "Union-Find" | "State-Machine" | "...";
    complexity: string;
    codeSnippet: string;
  }[];
  
  // Section 6: Demo
  demo: {
    url: string;          // Demo 地址
    type: "iframe" | "external";
    autoPlay?: boolean;
  };
  
  // Section 7: 元数据
  tags: string[];
  techStack: string[];
  stats: { views: number; likes: number; };
  published: boolean;
}
```

**关键决策**：版主不再"写文章"，而是"填写 Schema"。页面由 Schema 自动生成。这是 AI 流水线化的前提。

### 5.4 Demo（最小实现）— 体验闭环

Demo 的设计原则：**5 秒进入，30 秒理解**。

```
Demo 页面结构：
├─ Canvas 区域（占首屏 60%）
├─ 规则条（3 条以内，可折叠）
├─ 控制按钮：开始 / 重置 / 自动演示
└─ 参数面板（可选，默认折叠）

禁止：
× 复杂菜单
× 多级导航
×  lengthy 教程
× 登录/注册门槛
```

**V2 技术决策**：Demo 独立部署（demo.ovokit.com 或子路径），与主站通过 iframe + postMessage 通信。

### 5.5 Algorithm（算法库）— 实现参考

算法库与 Play 解耦，可被多个 Play 引用。

```typescript
interface AlgorithmSchema {
  id: string;
  name: string;           // 例如："Flood Fill 匹配检测"
  category: "匹配" | "寻路" | "生成" | "判定" | "状态机";
  
  // 多语言实现
  implementations: {
    language: "typescript" | "javascript" | "python" | "csharp" | "gdscript";
    code: string;
    framework?: "pixijs" | "phaser" | "unity" | "godot" | "none";
  }[];
  
  // 复杂度分析
  complexity: {
    time: string;
    space: string;
  };
  
  // 应用场景
  usedBy: string[];       // 引用的 Play slug 列表
}
```

## 六、母型页面：七段式标准结构

V2 要求每个母型页面（/archetypes/[key]）遵循固定的七段式结构。这是降低用户认知成本、提升 AI 生成效率的关键标准化。

### Section 1: 母型认知锚点（首屏上部）

目标：**5 秒内让用户知道"这是什么玩法"**。

内容：
- 母型名称 + 一句话定义
- 3 个核心特征标签
- 难度指示器
- 学习目标（"学完你能做出什么"）

### Section 2: 30 秒试玩（首屏核心）

目标：**让用户立刻体验到"好玩"**。

内容：
- 嵌入式 Demo Canvas（占首屏 50%+ 宽度）
- 极简控制：开始 / 重置 / 自动演示
- 悬浮规则提示（不超过 3 条）

### Section 3: 系统拆解

目标：**回答"为什么好玩"**。

内容：
- 玩家目标
- 核心决策点
- 反馈机制
- 循环结构
- 推荐：Mermaid 流程图或手绘 SVG 流程图

### Section 4: 玩法公式库

目标：**启发组合创新**。

内容：
- 基础公式：母型 = A + B + C
- 组合示例：
  - 消除 + 成长 = 养成消除
  - 消除 + 战斗 = Puzzle RPG
  - 消除 + Roguelike = Roguelike Match
- "尝试这个组合"入口

### Section 5: 高级设计（默认折叠）

目标：**让有经验的开发者避免踩坑**。

内容：
- 数值陷阱
- 算法陷阱
- 体验陷阱
- 真实案例（"XX 游戏在这里做错了"）

### Section 6: 实现参考

目标：**给程序员直接可用的代码**。

内容：
- 状态机图
- 核心算法
- 代码片段（带语法高亮）
- 复杂度分析

### Section 7: Demo 源码与相关玩法

目标：**闭环学习路径**。

内容：
- Demo 源码链接（GitHub / CodeSandbox / StackBlitz）
- 相关 Play 推荐（同母型 + 跨母型组合）
- "下一步学习"指引

## 七、内容生产机制：结构化 + AI 流水线

### 7.1 当前痛点

版主当前发帖流程：
1. 手写 meta.json（易错）
2. 手写 article.mdx（自由度过高，结构不统一）
3. 手动上传图片/视频
4. 手动配置 Demo

**问题**：生产一个 Play 需要 2-4 小时，且质量依赖版主个人能力。无法规模化。

### 7.2 V2 目标：15 分钟生产一个高质量 Play

通过"结构化表单 + AI 补全"实现：

```
版主输入（5 分钟）
    │
    ├─ 玩法名称
    ├─ 选择母型
    ├─ 难度
    ├─ 核心循环（一句话）
    ├─ 3 个设计要点（bullet points）
    ├─ 1 个代码片段
    └─ Demo URL
    │
    ▼
AI Agent 补全（10 分钟，自动）
    │
    ├─ Agent A: 生成完整拆解文案
    ├─ Agent B: 生成 SEO 标题/描述/FAQ
    ├─ Agent C: 生成 SVG 封面图
    ├─ Agent D: 提取算法标签
    └─ Agent E: 生成玩法公式组合建议
    │
    ▼
自动发布
```

### 7.3 Play Schema 2.0（版主表单版）

```typescript
interface PlayFormInput {
  // 必填（版主输入）
  title: string;
  archetype: PlayArchetypeKey;
  difficulty: PlayDifficulty;
  summary: string;           // 30秒简介
  coreLoop: string;          // 核心循环
  rules: string[];           // 3-5条规则
  keyCode: {                 // 1个关键代码
    title: string;
    language: string;
    code: string;
  };
  demoUrl?: string;
  
  // 选填
  tags?: string[];
  techStack?: string[];
  coverImage?: File;
}

// AI 自动生成（基于 Schema）
interface PlayGenerated {
  breakdown: BreakdownSection[];
  seoTitle: string;
  seoDescription: string;
  faq: { q: string; a: string }[];
  coverSvg: string;
  algorithmTags: string[];
  comboSuggestions: string[];
  pitfalls: PitfallSection[];
}
```

### 7.4 AI Agent 流水线（V2.1 实现）

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Agent A    │───→│  Agent B    │───→│  Agent C    │
│  内容生成    │    │  SEO 优化    │    │  封面生成    │
└─────────────┘    └─────────────┘    └─────────────┘
       │
       ▼
┌─────────────┐    ┌─────────────┐
│  Agent D    │───→│  Agent E    │
│  算法标注    │    │  组合推荐    │
└─────────────┘    └─────────────┘
```

**Agent A（内容生成）**：基于版主输入的 bullets，生成完整的系统拆解文案。

**Agent B（SEO 优化）**：生成 5 个标题变体、3 个描述变体、FAQ 问答对。

**Agent C（封面生成）**：基于玩法名称和母型，调用 RoughJS SVG 生成器生成封面图。

**Agent D（算法标注）**：分析代码片段，自动标注使用的算法（DFS/BFS/状态机等）。

**Agent E（组合推荐）**：基于母型数据库，推荐"玩法组合"方向。

### 7.5 内容质量控制

三层质检机制：

1. **Schema 校验**：表单提交时 Zod 强类型校验
2. **AI 自检**：Agent A 生成后，自动检查"是否覆盖了核心循环"
3. **版主终审**：生成预览页面，版主确认后发布（published: true）

## 八、用户交互体系：学习闭环

### 8.1 当前状态

当前 OVOKIT 是"只读站"：
- 用户只能浏览
- 无法互动
- 无法收藏
- 无法记录学习进度

### 8.2 V2 交互升级（分阶段）

#### Phase 1: 匿名互动（无需账号）

| 功能 | 实现方式 | 价值 |
|------|---------|------|
| 点赞 | localStorage + API，防重复 | 内容质量信号 |
| 浏览计数 | 已完成 ✅ | 热门排序依据 |
| 搜索历史 | localStorage | 个性化体验 |

#### Phase 2: 轻量账号（可选登录）

| 功能 | 说明 |
|------|------|
| 收藏玩法 |  localStorage 同步到账号 |
| 学习进度 | "已浏览 / 已试玩 / 已深入" 标记 |
| 笔记 | 在代码片段旁添加个人笔记 |

#### Phase 3: 学习路径

```
用户行为 → 推荐算法 → 学习路径

例如：
用户浏览了 "Match-3" → 推荐 "Puzzle RPG"（Match + Combat）
用户试玩了 Demo → 推荐 "实现参考"文章
用户收藏了 3 个消除类 → 生成 "消除玩法专题学习路径"
```

### 8.3 关联玩法推荐算法（V2.2）

基于母型的关联规则：

```
关联强度 = 同母型(0.5) + 共享算法(0.3) + 组合关系(0.2)

例如：
Match-3 → Puzzle RPG（组合关系 + 共享匹配算法）
Match-3 → Candy Crush（同母型）
Tower Defense → Idle（共享"放置"机制）
```

### 9.3 关键技术决策

| 决策 | 当前 | V2 | 理由 |
|------|------|-----|------|
| 内容存储 | 文件系统 | 文件系统 + [未来] SQLite | 文件系统适合版本控制，SQLite 适合查询和统计 |
| Demo 部署 | 嵌入主站 | 独立子域名 | 解耦、独立迭代、性能隔离 |
| 图片处理 | 原图直存 | 原图 + [未来] WebP 自动转换 | 减少带宽 |
| 搜索 | 客户端过滤 | [未来] 服务端搜索 + 拼音 | 支持大量内容后的性能 |
| 缓存 | 无 | ISR (60s) + CDN | 已部分实现 ✅ |

### 9.4 Demo 系统详细设计

**为什么必须独立？**

1. **性能隔离**：Demo 的 Canvas 渲染不影响主站 TTI
2. **技术自由**：Demo 可用 PixiJS/Phaser，主站保持 React
3. **独立迭代**：Demo 更新不需要重新部署主站
4. **SEO 安全**：iframe 内容不影响主站 Core Web Vitals

**通信协议**：

```typescript
// 主站 → Demo
interface DemoCommand {
  type: "demo:start" | "demo:reset" | "demo:setParam";
  payload?: unknown;
}

// Demo → 主站
interface DemoEvent {
  type: "demo:ready" | "demo:score" | "demo:gameOver";
  payload?: unknown;
}
```

## 十四、结语

OVOKIT V2 的战略本质是：**把"游戏玩法知识"从散落在博客、论坛、视频中的非结构化信息，转化为标准化、可组合、可验证的知识库。**

在 2026 年的 AI 时代，信息不再稀缺，稀缺的是**结构化、可行动、可验证的知识**。OVOKIT 的价值不在于有多少篇文章，而在于：

> **当开发者搜索"match 3 mechanics"时，OVOKIT 能在 30 秒内让他理解核心循环，3 分钟内看到实现方案，10 分钟内运行 Demo 验证。**

这是导航站做不到、教程博客做不到、AI 聊天机器人也做不到的事情——因为后者没有**体系化的知识架构**和**可试玩的验证闭环**。

**下一步行动**：落地 Phase 1（Schema 定义 + 结构化表单 + 母型标准化），为 AI 流水线打下标准化基础。
