## 1. 项目目的（Why）

### 1.1 核心问题

玩法研发与学习经常遇到三个痛点：

1. 经验难沉淀：玩法拆解与关键代码零散在笔记/聊天记录/仓库片段中，不可检索、不可复用。
2. 学习成本高：缺少“结构化拆解（交互→规则→判定→数据结构→关键算法）”的统一模板。
3. Demo 难共享：演示经常和页面强耦合，导致移动端输入、滚动、样式互相干扰，不利于传播与复用。

### 1.2 目标产出

- 把“玩法”拆成可复用的结构化信息：拆解 / 关键代码 / Demo / 文章（可检索、可复用、可持续迭代）。
- 提供轻量的本地内容发布流程（MVP），让内容可持续增长。
- 为后续更多玩法实例提供底座：嵌入式 Viewer（iframe）+ Demo/API（服务端为执行中心）。

## 2. 站点介绍（What）

### 2.1 一句话介绍

OVOKIT 是一个面向游戏开发者的“玩法技术实现分享站”，把每个玩法沉淀成结构化内容 + 可嵌入演示，从而支持学习、复用与传播。

### 2.2 目标用户

- 主要：玩法程序/技术策划/客户端工程师（需要快速复现玩法核心交互与判定）
- 次要：新人学习（需要“母型玩法”目录与入门路径）
- 扩展：团队内部知识库（需要可检索与统一结构）

### 2.3 内容形态

- 玩法帖子（Play）：标题/副标题/标签/难度/技术栈/核心点/拆解/代码片段/Demo/文章
- 母型玩法（Archetype）：固定 12 项，作为玩法目录与学习路径的“锚点”
- Demo/Viewer：用于隔离演示交互（建议 iframe 嵌入），避免影响站点页面滚动与样式

## 4. 当前实现进度（As-is）

### 4.1 前端站点（Next.js App Router）

- 已有：站点壳（TopNav/BottomNav/MenuDrawer）、响应式布局
- 已有：玩法信息流卡片、标签/难度/技术栈等展示
- 已有：筛选与搜索（`?group=...&cat=...&q=...`，并兼容旧 key 映射）
- 已有：玩法详情页结构（拆解/代码片段/Demo 区/文章区）
- 已有：Route Group 隔离
  - `(site)`：常规页面
  - `(embed)`：嵌入页（避免交互/滚动干扰）

### 4.2 内容体系（本地文件）

- 玩法内容源：`content/plays/<slug>/{meta.json,article.mdx}`
- 母型玩法内容源：`content/archetypes/<key>/meta.json`
- 读取方式：服务端通过 `node:fs` 读取（适合 Node 服务器 / 本地开发；部署到无持久化磁盘的 Serverless 需要额外方案，见“上线缺口”）

### 4.3 版主模式（站内发布 / MVP）

- 登录：`POST /api/mod/login` 校验环境变量 `MOD_PASSWORD`，写入 httpOnly cookie（`ovokit_mod=1`）
- 状态：`GET /api/mod/me` 返回是否版主
- 登出：`POST /api/mod/logout` 清理 cookie
- 发布：`POST /api/mod/plays`
  - 写入：`content/plays/<slug>/meta.json` 与 `article.mdx`
  - 支持上传：封面（≤5MB）、宽封面（≤5MB）、演示视频（≤50MB）
  - 资源落盘：`public/plays/<slug>/{cover.*,cover-wide.*,demo.*}`
  - slug 校验：仅允许小写字母/数字/连字符

> 结论：版主发布流程在“本地/有持久化磁盘的 Node 服务器”可用；在典型 Serverless（如只读文件系统/无持久化）不可用，需要改造。

### 4.4 Demo / Viewer（演示解耦）

- BlockKit：Editor（产出数据）与 Viewer（展示/可嵌入）拆分
- 嵌入建议：站内展示优先 iframe（避免移动端输入/滚动竞争）

### 4.5 算法/Demo 后端（Fastify）

- 位置：`server/*`（Fastify + Zod）
- 用途：承载最小规则试玩/算法示例（服务端为执行中心）

## 5. 上线定义：你要“上线什么版本”

上线并不是一个开关，而是“对外承诺的能力集合”。建议先选一个上线级别：

### 5.1 上线级别 A：只读公开站（推荐先达成）

对外承诺：内容可浏览、可阅读、演示可打开；不承诺站内发布与实时互动数据。

适用：对外展示 / 冷启动 / 内容量不大 / 先验证信息架构与内容模板。

### 5.2 上线级别 B：带站内发布（不建议在 Serverless 上直接上线）

对外承诺：站内可登录发布内容、上传封面/视频，并可长期保存。

适用：有持久化存储（数据库/对象存储/Git 工作流）与更完整运维能力后再上。

## 6. 上线需求对齐（P0/P1/P2）

下面按“上线级别 A（只读公开站）”为基准给出清单；若要上线级别 B，会额外列出必须项。

### 6.1 P0（上线级别 A：必须满足）

**内容与信息架构**

- 首页 → 详情 → 母型玩法路径可用，且主要入口不指向“占位/死链”
- 玩法详情页至少做到“文章可读”（即使不做完整 MDX，也要可读、可复制、无明显乱码/断行）

**稳定性**

- `pnpm build` 可通过（作为上线最低门槛）
- 关键路由 Smoke Test：`/`、`/play/<slug>`、`/archetypes`、`/archetypes/<key>`、`/embed/*` 能正常渲染或给出清晰 404

**安全与暴露面**

- 线上默认不暴露“写入本地文件”的能力：
  - 若部署环境无法保证持久化与权限隔离，建议直接禁用 `/mod` 与 `/api/mod/*`（或限制在内网/BasicAuth/白名单）
  - 至少需要防爆破（简单限流）与更安全 cookie 配置（见 6.4）

**基础 SEO**

- 每个玩法详情有可索引的标题与描述（`metadata` / `<title>` / `<meta name="description">`）
- `robots.txt`、`sitemap.xml`（哪怕最简）具备（否则收录与分享效果差）

### 6.2 P1（上线后 1–2 周内补齐，强烈建议）

- 文章渲染从“原文展示/占位”升级为可用的 Markdown/MDX 渲染（并明确 XSS 策略）
- 404 / 错误页体验：缺 slug、缺内容文件时返回统一的友好页面
- 基础统计口径：将“likes/views 占位字段”要么隐藏要么落地（避免误导）
- 内容质量模板：为每个玩法提供最小必填字段校验（例如拆解至少 1 段、核心点至少 3 条等）

### 6.3 P2（体验增强，可排期）

- 更细的筛选组合、排序策略、标签聚合页
- 站内搜索体验：高亮、空结果引导、拼写容错
- Demo/Viewer 的模板库扩充与内容联动（玩法帖可直接关联某个 embed 模板）

### 6.4 若要上线级别 B（带站内发布），额外必须项

**存储与部署形态**

- 不能依赖“写入 `content/` 与 `public/` 本地文件系统”作为长期存储（典型 Serverless 不可用）
- 必须选定并实现一种持久化方案：
  - Git 内容工作流（PR 合并后发布）
  - 数据库 + 对象存储（封面/视频走 OSS/S3）
  - 自建 Node 服务器 + 持久化磁盘（并处理备份/迁移）

**权限与安全**

- `MOD_PASSWORD` 登录接口需要限流/防爆破
- cookie 建议补齐 `secure`（HTTPS）与更明确的生命周期策略
- 上传文件需要更严格的 MIME/内容校验与扫描策略（至少白名单 + 尺寸限制 + 路径隔离）

## 7. 风险与结论（是否符合上线）

### 7.1 主要风险（当前已知）

- **发布能力的部署风险**：`/api/mod/plays` 会写入仓库目录（`content/*`、`public/*`）。若部署到无持久化文件系统或只读环境，将导致发布失败或数据丢失。
- **安全风险**：登录为单口令 + cookie，缺少限流；若暴露公网存在被爆破的风险。
- **内容渲染风险**：文章目前是“原文占位/不解析”，对外上线可能显得不完整（取决于你对“上线”的定义）。

### 7.2 上线判断（建议）

- **如果上线目标是“级别 A：只读公开站”**：整体结构与能力已接近可上线，但建议至少补齐 P0 的“文章可读 + 基础 SEO + 线上禁用/限制发布入口”。
- **如果上线目标是“级别 B：带站内发布”**：目前不建议直接对外上线，需先完成“持久化方案 + 权限安全加固”。

## 11. P0（只读站）上线：内容数量与 AdSense 审核口径

### 11.1 先说结论：没有“官方固定篇数”，但需要“足够的可读内容”

Google/AdSense 官方并不会给出“必须 X 篇文章才能过审”的固定数字；审核更关注：

- 网站是否已完整上线、可正常导航访问（不是“模板/占位/施工中”状态）
- 页面是否有足够的原创、可读文本（避免“内容过少/页面空/大量重复”）
- 是否具备基本的站点信息与合规页面（通常至少要有：关于/联系/隐私政策；以及清晰的站点目的与导航）

参考（官方/权威入口，供后续对齐）：  
- AdSense Program policies：https://support.google.com/adsense/answer/48182  
- 关于“有价值的库存（valuable inventory）/内容不足”等常见拒绝原因：https://support.google.com/adsense/answer/10015918

### 11.2 OVOKIT 的“可执行”上线内容标准（建议作为内部验收线）

为了提高过审概率，并避免“内容过少/占位”的主观风险，建议把 P0 只读上线的内容标准定为：

**建议最低（上线前）**

- 玩法帖子（Play）：至少 `20` 篇达到“可审查就绪”
  - 建议口径：每篇 `article.mdx` 文本长度 `>=800` 字符（越多越好，且要结构化：标题/小节/列表/例子）
  - 玩法详情页无明显“占位/TODO/只有几句话”的情况
- 母型玩法（Archetype）：`12` 个固定项全部可读（已有结构化内容；但需确认前端页面实际展示到位）
- 站点静态页：至少补齐 `关于` / `联系` / `隐私政策`（且内容非占位）

**现状对比（2026-02-03）**

- Play：`30/30` 文章 `>=600` 字符，`>=800` 达到 `20/30`；封面 `30/30` 具备；Demo（iframe/video）已接入 `6/30`（其余可先用母型最小示例回退）。
- 母型玩法：`12/12` 可读，且每个母型页具备最小可试玩 Demo。
- 站点静态页：已补齐 `关于/联系/隐私政策/使用条款`，并确保导航可达。

> 备注：上面的字符阈值是“工程团队内部的可执行标准”，不是 Google 的硬性规则；目的是把主观审查问题变成可量化、可追踪的内容任务。
